# PokerTH Protobuf bindings

Auto-generated JavaScript classes for the PokerTH wire protocol, plus a
friendly facade for the messages the client actually uses.

## Why

The original `pokerth.js` carries a small hand-written Protobuf
encoder/decoder that hard-codes every field number inline. That worked but
is brittle: any change to `pokerth.proto` upstream silently breaks
encoding. This module replaces it with the official protobuf.js runtime
and a static class bundle generated from the schema.

## Files

| File | Purpose | Size |
|---|---|---|
| `pokerth.proto` | Copy of the upstream schema (PokerTH v2.0.7) | ~28 KB |
| `protobuf-minimal.mjs` | protobufjs v8 minimal runtime, ESM-wrapped | ~24 KB |
| `pokerth-bundle.mjs` | Static module generated by `pbjs` | ~470 KB unminified (~50 KB gzipped) |
| `index.mjs` | Friendly facade with helpers (`buildInit`, `decode`, …) | ~6 KB |

## Usage

```js
import { buildInit, LoginType } from '/proto/index.mjs';

const bytes = buildInit({
    nick: 'Alice',
    major: 5,
    minor: 1,
    login: LoginType.guestLogin,
});
ws.send(bytes);
```

See `index.mjs` for the full list of helpers. Each helper is a drop-in
replacement for the corresponding hand-written builder in `pokerth.js`.

## Equivalence guarantee

Before this module is wired in, the produced bytes have been verified
to match exactly the captures from the proxy logs. For example, the
unauthenticated InitMessage for `Guest90134` produces:

```
08021a190a040805100110868088082802320a47756573743930313334
```

…identical to what the previous manual encoder produced.

## Regenerating the bundle

If the upstream `pokerth.proto` evolves, run:

```bash
npm install
npm run build:proto
```

That re-runs `pbjs` with the same flags used to produce the current
bundle (kept reproducible in `package.json`).
